# IFLOW.md - 核心工作规则

## 1. 元约束（Meta Constraints）

> 违反任意一条即视为任务失败，无任何修复机会

- 100% 中文回复（zh-CN，简体，技术术语可保留英文）
- 100% 先调用子代理（无例外，主上下文只做路由）
- 100% 通过基础安全检查（无恶意代码、无敏感数据泄露）
- 100% 遵循编程规范与工程原则（SOLID、KISS、DRY、YAGNI）
- 100% 符合工程原则（SOLID、KISS、DRY、YAGNI）

## 3. 子代理工作模板（复杂度下沉）

1. 任务拆解：输出「用户故事 → 技术任务」映射表
2. 工具链：在子代理内部按顺序调用 MCP 工具（search → urls_fetch → code/write）
3. 代码评审：必须执行「静态扫描 + 单元测试 + 性能剖析」三重门禁
4. 结果验证：对照「CR 检查单」与「工程原则检查单」双签字后方可返回主上下文
5. 文档产出：必须输出「变更日志 + 接口契约 + 设计文档 + 测试报告」四大件
6. 交付格式：必须符合「输出格式契约」要求
7. 安全检查：必须通过「安全检查单」零容忍检测
8. 编程规范：必须符合「编程规范」要求
9. 工程原则：必须通过「工程原则检查单」审核
10. 路由反馈：必须附带「路由决策依据 + 子代理工作日志」
11. 任务总结：必须输出「任务总结 + 经验教训 + 改进建议」
12. 后续计划：必须包含「后续任务清单 + 优先级排序 + 预估工时」
13. 用户确认：必须等待主上下文「用户确认」后方可结束
14. 版本控制：必须提交至「代码仓库 + 文档库 + 任务管理系统」
15. 质量保障：必须通过「QA 团队」最终验收
16. 迭代计划：必须制定「下一版本迭代计划 + 时间表」

## 4. 编程规范（强制内嵌）

| 维度 | 规约                                        |
|----|-------------------------------------------|
| 命名 | 统一使用英文驼峰/蛇式，禁止拼音；常量全大写加下划线                |
| 函数 | 单行长度 ≤ 80；圈复杂度 ≤ 5；必须纯函数优先                |
| 类  | 单文件单类；职责>1 立即拆分（SRP）                      |
| 注释 | 公共 API 必须包行内文档（docstring）；业务代码「为什么」>「做什么」 |
| 异常 | 禁止裸 `except:`；自定义异常继承自 `DomainException`  |
| 测试 | 新增代码覆盖率 ≥ 90%；TDD 红线→绿线→重构流程              |
| 文档 | 函数参数文档化；类文档化；接口文档化                        |
| 配置 | 禁止硬编码配置；配置项必须使用环境变量；配置项必须使用配置管理器          |
| 版本 | 语义化版本号；每次提交必须更新 `CHANGELOG.md`            |    
| 测试 | 新增代码覆盖率 ≥ 90%；TDD 红线→绿线→重构流程              |
| 架构 | 模块化设计；无副作用函数；无状态函数；无副作用类；无状态类             |
